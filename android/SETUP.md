# Android app shell (Phase 2)

Native Kotlin/WebView shell for the Retirement Planning v10 system, per
`documentation/ANDROID_MOBILE_ENHANCEMENT_PLAN.md` §4 and Phase 2. This
directory is a Gradle project that packages the existing `src/` tree
(via Chaquopy's embedded CPython) and `frontend/` (verbatim, no build step)
into an APK, replacing PyWebView with a native WebView + a Kotlin
`JavascriptInterface` bridge.

## Status

**Source-scaffolded, not yet build-verified.** This was written in a
sandboxed dev container with no Android SDK, no Gradle wrapper distribution,
and no emulator/device available, so `./gradlew assembleDebug` has not been
run against it. Before relying on it:

1. Open `android/` in Android Studio (Giraffe+) — it will offer to generate
   the Gradle wrapper if missing.
2. Let it sync. The Chaquopy plugin needs a license-free Maven repo entry
   (already declared in `settings.gradle.kts`) and will download the Python
   pip dependencies (`numpy`, `openpyxl`, `reportlab`, `matplotlib`,
   `pillow`, `cryptography`) — this is the "Chaquopy dependency spike" the
   plan calls out as risk §7.1; expect to spend time here on wheel
   availability/version pinning before anything else works.
3. Build and run on an arm64 emulator or device (API 26+, though a modern
   API 29+ target is what's actually been designed for — see the Downloads
   note below).

## How the pieces fit together

- **`app/build.gradle.kts`** — Chaquopy's Python source set is rooted at the
  repo root (not `src/` directly) with `include("src/**", "reference_data/**",
  "system_config.csv")`, so the extracted layout on-device keeps `src` as a
  real importable package and `src/server/app_core.py`'s `BASE_DIR` (its own
  `__file__.resolve().parents[2]`) finds `reference_data/` and
  `system_config.csv` as siblings — the same relative layout the desktop
  checkout has. `tools/` is excluded: the subprocess build path that needs it
  never runs on Android.
- A `copyFrontendAssets` Gradle task copies `../frontend` into
  `app/src/main/assets/frontend` (wired into `preBuild`) so `frontend/` never
  needs to be duplicated by hand and stays byte-identical to what the desktop
  and server modes load.
- **`RetirementApplication.kt`** starts the embedded interpreter once
  (`Python.start(AndroidPlatform(this))`).
- **`MainActivity.kt`** hosts a single `WebView`, serves `assets/frontend/`
  through `WebViewAssetLoader` at `https://appassets.androidplatform.net/`,
  and wires a `WebChromeClient.onShowFileChooser` so the existing
  `<input type=file>` flows (YTD CSV import, `.rpx` import) work via
  `ACTION_OPEN_DOCUMENT` with no route changes.
- **`PythonBridge.kt`** is the `JavascriptInterface` (`window.AndroidBridge`
  in JS). Its `request()` method is void, not return-value: WebView's JS
  interface calls block the calling JS thread until the method returns, which
  would freeze the UI for the length of a workbook build. Work runs on a
  background executor and the result is delivered back into the page via
  `evaluateJavascript` instead.
- **`src/android_api.py`** (in the main repo, not under `android/`) is the
  Python side of that bridge — mirrors `src/desktop_api.py`'s `request()`
  against the same local route registry, but returns binary payloads as
  base64 (`_binary`) instead of shelling out to a desktop file-open command.
  `android_api.configure(workspace_root)` points `platform_runtime` at the
  app's private `filesDir` and flips `platform_runtime.is_mobile()` on —
  which is what makes the Phase 0 `BuildRunner` pick the in-process
  (threaded) build path automatically. No Android-specific build code was
  needed for that.
- **`frontend/js/android_bridge.js`** is the JS-side transport: a
  request-id/callback table standing in for the Promise PyWebView's `js_api`
  gives you for free, plus a `location.href` intercept for the two internal
  nav targets (`/`, `/system-configuration`) since `dashboard.js`/`admin.js`
  only special-case `window.pywebview` today. It is a no-op unless
  `window.AndroidBridge` exists, so it's safe to always load alongside
  `pywebview_bridge.js`.

## First run

There is no bundled "blank plan" template to keep in sync. `PythonBridge`
calls the existing `POST /api/plan-data/blank` route once on first launch
(guarded by a marker file in `filesDir/local_state/`) — the same route the
in-app "Start New Plan" button uses — so blank `input/*.csv` templates are
always generated by the one, already-tested code path
(`PlanDataFileService.start_blank_payload()`), never duplicated here.

## Downloads (known gap on API 26-28)

`PythonBridge.saveFile()` writes workbook/PDF/`.rpx` exports through
`MediaStore.Downloads`, which requires API 29+ (scoped storage). A legacy
fallback for API 26-28 is stubbed in using the pre-scoped-storage public
Downloads directory, but the runtime permission request flow for
`WRITE_EXTERNAL_STORAGE` (declared in the manifest, `maxSdkVersion=28`) is
**not wired up yet** — that's flagged as a TODO in `PythonBridge.kt` and
should be treated as a real gap, not a placeholder comment, until it's
tested against an API 26-28 device. Given this is a single-user household
app, targeting a modern device (API 29+) first and treating the legacy path
as a follow-on is a reasonable sequencing call.

## What's deliberately not here yet (Phase 3 per the plan)

- App icon / adaptive launcher graphics.
- ProGuard/R8 rules for a real release build (`isMinifyEnabled = false` for
  now — Chaquopy's reflection-based JS interface calls need care under R8
  that hasn't been validated).
- CI building the debug APK (plan §3.5) and an instrumented Espresso/
  UIAutomator smoke test (§3.6).
- WebView hardening review / `file://` disabling verification, and an
  Android backup-policy (`dataExtractionRules`) writeup (§3.4) — `allowBackup`
  is already set to `false` in the manifest as a first pass.
