# System Review & Refactor Plan

Generated: 2026-07-02

Scope: full-repository review of the retirement planning system (~60K lines of
Python across `src/`, ~5K lines of JS across `frontend/`, 142 pytest files)
covering legacy/dead code, oversized modules, test coverage, performance,
portability, maintainability, and usability. This document is the output of
that review and the implementation plan to act on it.

## 1. Executive summary

The codebase is functionally mature (550 passing tests per
`PROJECT_MANIFEST.md`) and has already been through deliberate cleanup passes
(see "Cleaned From Root" in the manifest). The remaining debt is concentrated
in five places:

1. **No safety net around the code** — zero CI, no lint/type-check config, no
   `conftest.py`/`pytest.ini` (tests don't even run from repo root without
   manually setting `PYTHONPATH`), and `pytest` itself isn't a declared
   dependency. Any refactor is currently unguarded.
2. **A handful of genuinely oversized modules** — `planning_engines.py` (2692
   lines), `sheets_summary.py` (2660 lines, with a single 1415-line function),
   `dashboard.js` (2483 lines, decomposition started and stalled), `data_io.py`
   (2304), `app_core.py` (2015, a god-module mixing auth/audit/IO/config).
3. **Real portability bugs**, not just theoretical risk — the PyInstaller
   build script's own success check is Windows-only and will misreport
   failure on Mac/Linux; a hardcoded personal Windows path/name is baked into
   the default build flow.
4. **A critical, untested security module** — `src/security.py` (redaction,
   constant-time token comparison, audit logging) has zero direct test
   coverage.
5. **Committed build output** — `output/js`/`output/css` are generated files
   tracked in git, already drifted from their `frontend/` source of truth.

None of this is "legacy" in the rot sense — most of what looks like legacy
code (dual sheet-naming, `roth_legacy_score`, `flask_compat.py`) turns out to
be either live domain terminology or a load-bearing shim with a misleading
name. The actual legacy/dead code is a small, well-defined list (Section 2).

## 3. Oversized modules / refactor targets

Ranked by impact (size × how central the module is):

1. **`src/reporting/sheets_summary.py`** (2660 lines) — `build_sheet4`
   (Asset Allocation) is a single **1415-line function** with 5+ nested
   closures. Split into module-level helper functions; the shared styling
   helpers it needs already exist in `workbook_common.py` (53 call sites
   elsewhere), so this is pure decomposition, not new infrastructure.
2. **`src/planning_engines.py`** (2692 lines, largest file in the repo) —
   **investigated during Phase 2c and deliberately left as one file.** The
   file carries literal `# ===== BEGIN/END <name>.py =====` markers showing
   it's the result of a prior, deliberate consolidation of 8 previously
   separate engine files (growth/inheritance/mortality/withdrawal/conversion/
   projection/monte_carlo/vectorized-monte-carlo engines). Six
   underscore-prefixed aliases defined near the top (`_ar`, `_aa`, `_we`,
   `_ce`, `_ie`, `_ge`) look like unused leftovers by local analysis, but are
   actually consumed by `src/projection_stages/deterministic_engine.py` via
   `from ..planning_engines import *` (which skips underscore names) followed
   by explicit `_legacy_pe._aa` etc. rebinding — that module's entire
   deterministic projection body (RMD computation, Roth conversion,
   withdrawal ordering, growth application, ~1600 lines) depends on these
   staying put. A first attempt at "cleaning up" these as dead code broke the
   engine at import time; caught immediately by a golden-master byte-diff of
   `project()`/`monte_carlo()` output before attempting any change. Splitting
   this file into separate modules would require carefully tracing every
   cross-section call first (the Monte Carlo section alone calls
   `sample_household_death_years` and `project()`, both defined in other
   sections) and risks reintroducing whatever circular-import problem the
   original consolidation likely solved. What Phase 2c actually did instead:
   removed genuinely-confirmed-dead duplicate `typing` imports and a
   redundant `_ar` re-import accumulated across the consolidation boundaries
   (17 caught by `ruff --select F401,F811 --fix`), one dead `end_yr` local
   variable, and one unused `.observability.observe` import — all verified
   safe via the same golden-master diff. Do not attempt a structural split of
   this file without a much deeper, dedicated investigation than a general
   refactor pass allows.
3. **`src/server/app_core.py`** (2015 lines) — a god-module mixing auth/cookie
   handling, audit logging, CSV migration, and config mutation. Split each
   concern into its own module, continuing the `server_services/` extraction
   pattern already used for `strategy_asset_service.py`, `spending_service.py`,
   etc.
4. **`src/server/plan_routes.py`** (987 lines) — the service-extraction effort
   (`test_126_service_extraction.py`) is complete for `workbook_routes.py`/
   `base_routes.py`/`admin_routes.py` but **not** for `plan_routes.py`, which
   still has ~90 local handler functions manipulating CSVs directly. Extract a
   `plan_service.py` to match the established pattern; drop the wildcard
   import from `app_core` in the same pass.
5. **`frontend/js/dashboard.js`** (2483 lines, ~810 top-level functions) — a
   decomposition into `frontend/js/modules/` was started
   (`phase3_module_manifest.js` declares the intended `extraction_order`) and
   then stalled — the manifest is a 9-line shell with no code actually moved.
   **Investigated during Phase 2d; physical extraction deliberately deferred.**
   Almost every function reads/writes a large set of shared top-level mutable
   state (`rows`, `dirty`, `activeStep`, `buildHistory`, `ytdData`, ...) — the
   same kind of cross-section coupling that made splitting `planning_engines.py`
   risky, except JS has no equivalent of `ruff`'s F821/F401 to catch a mistake,
   no bundler (extraction means separate `<script>` tags sharing one global
   namespace, order-dependent), and — until this phase — zero unit-test
   coverage of any kind (only Python tests asserting the file contains certain
   substrings, which don't verify behavior). Moving code under those
   conditions is a good way to introduce a silent runtime regression a user
   only discovers in the browser. What Phase 2d did instead: added the first
   JS test coverage in the repo (`tests/frontend/`, Node's built-in
   `node --test`, 24 tests) covering the handful of functions that genuinely
   are pure — `esc`, `norm`, `titleWord`, `stripUiLabelPrefix`,
   `formatAcronyms`, `fmtMoney`, `fmtPct`, `finiteOrNull`, `firstFinite`,
   `normalizeLanguageMode` — via a small `vm`-based harness
   (`tests/frontend/load_dashboard.mjs`) that runs the real production file
   against minimal DOM/timer stubs. This surfaced a real, previously-untested
   quirk: `finiteOrNull(NaN)` returns `0` instead of `null` (`String(NaN)` →
   `"NaN"` → stripped of non-numeric characters → `""` → `Number("")` → `0`,
   which passes `Number.isFinite`), which silently short-circuits
   `firstFinite(..., NaN, ...)` to `0` instead of skipping past the `NaN` to
   the next real value — documented as a test case, not fixed, since other
   code may depend on it and this wasn't the task at hand. Recommended
   sequencing for whoever picks up the actual extraction: grow this test
   file's coverage function-by-function *before* moving each one (turns each
   move into a verified refactor instead of a leap of faith), starting with
   the next-safest tier (functions that read but don't write the shared
   state) rather than attempting the stateful majority in one pass.
6. **`src/data_io.py`** (2304 lines), **`src/market_data.py`** (1741),
   **`src/projection_stages/deterministic_engine.py`** (1669),
   **`src/optimization.py`** (1450), **`src/reporting/workbook_builder.py`**
   (1248) — lower priority than 1-5, but candidates for the same treatment
   once tooling (Phase 0 below) is in place to make large diffs safe.

## 4. Testing: gaps and fixes

**Structural gaps (fix first — these make every other test change riskier):**

- No `conftest.py` anywhere; no `pytest.ini`/`pyproject.toml`/`setup.cfg`.
  Running `pytest tests/test_x.py` from repo root fails with
  `ModuleNotFoundError: No module named 'src'` unless `PYTHONPATH=.` is set
  manually. Every contributor either knows this tribal knowledge or is
  blocked immediately.
- `pytest` itself is not in `requirements.txt` (or any `requirements-dev.txt`,
  which doesn't exist) — the one documented "how to run tests" instruction
  silently assumes a global install.
- `ROOT = Path(__file__).resolve().parents[1]` is copy-pasted in 102 of 142
  test files; a `sample_config()` helper is independently duplicated across
  at least two files. No shared fixtures exist.
- 22 of 142 files still use `unittest.TestCase`, the other 120 use bare
  pytest functions — two coexisting styles.
- No `@pytest.mark` usage anywhere, so there's no fast/slow split — at least
  one test class shells out to `tools/build_workbook.py` as a subprocess
  inside what's otherwise a pytest unit-test file.
- **9 test files read or regenerate the real, git-tracked
  `output/retirement_plan.xlsx` directly** rather than building into
  `tmp_path` — a genuine hermeticity problem (results depend on run order and
  prior on-disk state, and a test run mutates a version-controlled file).
- Test file numbering has collided (two `test_105_*`, two `test_106_*`, two
  `test_107_*`, `test_118_*`, `test_142_*`, `test_143_*`, `test_146_*`,
  `test_147_*` each exist twice under different suffixes) and there's no
  index mapping "tests for module X" to a filename — discovery is grep-only.

**Coverage gaps (by direct reference count, not just end-to-end workbook
tests):**

- **`src/security.py` — zero direct test references.** This is the module
  implementing `redact_secret`, `redact_text`, `constant_time_token_ok`
  (HMAC-based auth comparison), and `append_audit_event`. It is the single
  highest-priority testing gap in the repo.
- `src/secrets_store.py` — one incidental reference (an unrelated test
  imports two of its symbols in passing); no dedicated test file.
- `src/optimization.py` (1450 lines) — only referenced by early,
  allocation-UI-era tests (`test_4_*`–`test_15_*`, `test_29_*`); nothing
  after `test_29` exercises it directly.
- `src/market_data.py` (1741 lines) — only 6 files reference it directly.
- Frontend JS: no JS test framework/runner at all. `tools/validate_clean_overlay.py`
  runs `node --check` (syntax only) on 3 of ~10 JS files. Actual behavior
  verification is via Python tests that `.read_text()` the JS and assert
  substrings exist (`test_80_detailed_results_ui.py`,
  `test_ui_responsibility_split.py`) — this catches accidental deletions but
  cannot catch logic bugs.

## 5. Performance

- `app_core.py` writes a fresh copy of `system_config.csv` to disk **per
  request** for subprocess-based builds (`_request_system_config_csv`,
  `_make_request_system_config_csv_for`) — redundant I/O; pass config
  in-memory or cache the snapshot.
- Many `app_core.py` handlers re-read/re-write plan CSVs via
  `_csv_read_rows`/`_write_csv_rows_file` per call with no per-request
  caching.
- `data_io.py:parse_client` — a large, central transformation function,
  recomputed in full on every load with no memoization despite being called
  repeatedly per plan/report generation.
- `desktop_api.py:167` uses the deprecated, race-prone `tempfile.mktemp()`
  instead of `NamedTemporaryFile`.
- `enterprise_pdf.py:192` writes chart images to a **fixed filename**
  (`nw_chart.png`) in the shared temp directory instead of a unique
  mkstemp-generated path — a real collision risk if two report builds run
  concurrently (multi-user server mode, or two exports at once).
- `workbook_common.py`'s `auto_fit_columns`/`optimize_workbook_layout` do a
  full cell-by-cell scan (including merged-range checks) per sheet, run once
  per sheet (~20 sheets per workbook). Not a problem at current data volumes;
  worth flagging if projection horizons or sheet count grow substantially.

## 6. Portability

- **`build.py:45`** hardcodes the PyInstaller output path as
  `dist/retirement_planner/retirement_planner.exe`. On Linux/Mac, PyInstaller
  produces an extension-less binary, so the build script's own
  success-check will report a false "BUILD FAILED" after a successful build.
  This is a real bug, not a theoretical one, given the app claims cross-platform
  desktop packaging via pywebview.
- **`tools/backup_to_onedrive.py`** hardcodes a Windows-only OneDrive path
  with a real person's name and company (see Section 2) and is called
  unconditionally from `build.py` — breaks the build on any other machine/OS
  unless silently swallowed.
- **`desktop_api.py:173-174`** — `os.startfile(...)` (Windows-only) wrapped
  in a bare `except Exception` that falls back to
  `subprocess.Popen(["explorer", str(tmp)])` (also Windows-only). On
  Mac/Linux this either silently does nothing or raises an uncaught
  `FileNotFoundError`.
- No `sys.platform`/`os.name` branching exists for pywebview backend
  selection anywhere in `main.py`/`src/` — it relies entirely on pywebview's
  own auto-detection, which is untested per-OS in this repo (no CI, so no
  matrix build either).
- Documentation (`documentation/CLAUDE.md`, launcher docs) still describes
  Windows-only launchers (`BUILD.bat`, `START_APP.bat`) and a
  `C:\RetirementPlanning\...` working path as if that's the only supported
  environment.

## 7. Security

- `src/security.py` is untested (see Section 4) — the highest-priority item
  in this whole review given it implements auth-token comparison and audit
  logging.
- `src/secrets_store.py`'s `encryption_status`/`require_secure_master_key`
  are thin (a handful of lines) — worth explicitly confirming these aren't
  effectively no-ops before relying on them as a security boundary.
- No secrets are actually committed to git — confirmed via `git ls-files`
  that `local_state/`, `saved_plans/`, and most of `data/` never leaked; the
  only tracked file from those directories is a trivial preferences JSON.
  `system_config.csv` is schema/defaults only, with API-key fields explicitly
  blanked in the release package.
- The `tools/backup_to_onedrive.py` hardcoded name is a minor PII leak in
  source (a real person's name/company), not a credential leak — still worth
  fixing alongside the portability fix in Section 6.
- No `eval`/`exec`/`pickle`/`shell=True` usage found anywhere in `src/`; all
  subprocess calls use list-form arguments. This part of the codebase is
  already following good practice.

## 8. Maintainability infrastructure (the actual root cause)

Most findings above are symptoms of one thing: **there is no automated gate
on this codebase.** Concretely, the repo has:

- No CI/CD (`.github/workflows` doesn't exist, no other CI config found).
- No linter or formatter config (no `.flake8`, `.pylintrc`, `ruff.toml`,
  `.pre-commit-config.yaml`).
- No type-checker config, despite 89 of 105 `src/` files already using
  `from __future__ import annotations` (the codebase is halfway to being
  typed and getting no benefit from it).
- No `pyproject.toml`/`setup.py` — packaging metadata lives only in
  `retirement_planner.spec` and `requirements.txt`.
- No dependency lockfile — `requirements.txt` has loose upper bounds only.

This is why `output/` drifted, why `build.py` has a platform-specific bug
nobody caught, and why `security.py` has no tests — nothing runs
automatically to catch any of it.

## 9. Usability enhancements

- `frontend/index.html` has only 2 `aria-*`/`role` attributes total;
  `admin.html` has none. A basic accessibility pass (form labels, landmark
  roles, focus management on modals) is overdue.
- The migration from native `confirm()` to the in-app `showInAppConfirm`
  (already used at 35 call sites) is incomplete — two call sites in
  `dashboard.js` (lines 490, 1951) still use the native dialog, producing an
  inconsistent UX.
- Responsive design is reasonably covered (14 `@media` queries across three
  breakpoints) — not a priority.
- Suggested new capability: a `CONTRIBUTING.md` with the actual dev setup
  (`pip install -r requirements.txt -r requirements-dev.txt`, how to run
  `pytest`, how to run the lint/type-check once added) — currently
  undocumented outside tribal knowledge.

## 10. Implementation plan

Phased so each phase is independently shippable and safe to review. Phase 0
is a prerequisite for everything after it — do not start large refactors
(Phase 2) before it lands.

### Phase 0 — Safety net (do this first)
- Add `pyproject.toml` (or `pytest.ini`) with `pythonpath = ["."]` /
  `testpaths = ["tests"]` so `pytest` runs from repo root with no manual
  `PYTHONPATH`.
- Add a root `conftest.py` with shared fixtures: `ROOT`, `sample_config()`,
  a `tmp_workbook()` fixture that builds into `tmp_path` instead of
  `output/`. Migrate the 9 tests that read the real `output/retirement_plan.xlsx`
  to use it.
- Add `requirements-dev.txt` (pytest, pytest-cov, ruff, mypy at minimum).
- Add a GitHub Actions workflow: run `pytest`, `ruff check`, and a workbook
  smoke build on every PR.
- Add `@pytest.mark.slow` to the test(s) that shell out to
  `tools/build_workbook.py`; keep the default `pytest` run fast.
- Write the missing `security.py` unit tests (redaction, constant-time
  compare, audit log) before touching anything else security-adjacent.

### Phase 1 — Delete legacy/dead code and fix repo hygiene
- Remove `output/js`, `output/css` (and the rest of `output/`) from git
  tracking; add `output/` to `.gitignore`.
- Delete `compare_scenarios`/`sensitivity_grid`/`tornado_data` (or wire them
  up — confirm intent first), the two orphaned `tools/*.js` files, and the
  stale `desktop.ini` leftovers.
- Fix the stale "no `.git/`" claim in `documentation/CLAUDE.md`.
- Parameterize or remove `tools/backup_to_onedrive.py` from the default
  `build.py` flow.
- Consolidate the two CSV-row-purge implementations in `app_core.py` into
  one helper; evaluate retiring the purge entirely if the migration window
  has passed.
- Rename `flask_compat.py` → a name that doesn't imply it's removable.
- Rename `dashboard_roadmap11.js`/`dashboard_roadmap12.js` to
  intention-revealing names.
- Finish the `confirm()` → `showInAppConfirm` migration in `dashboard.js`.
- Drop the wildcard import in `plan_routes.py`.

### Phase 2 — Decompose the oversized modules
In priority order from Section 3: `sheets_summary.py`'s `build_sheet4`,
`planning_engines.py`, `app_core.py`, `plan_routes.py` (finish the service
extraction with a `plan_service.py`), then resume the stalled `dashboard.js`
→ `frontend/js/modules/` extraction per its own manifest. Each split should
land as its own PR with before/after test runs, since Phase 0's CI is what
makes this safe.

### Phase 3 — Performance fixes
Fix `tempfile.mktemp()` and the fixed-filename PDF chart path (both are
correctness bugs as much as performance ones — race conditions), then
address the per-request CSV disk-copy and `parse_client` recomputation.

### Phase 4 — Portability fixes
Fix `build.py`'s platform-specific success check, fix `desktop_api.py`'s
open-file fallback to branch by platform (`open` on Mac, `xdg-open` on
Linux), and finish removing the hardcoded OneDrive path (overlaps Phase 1).
Once Phase 0's CI exists, consider a Linux/Mac job in the matrix even if the
desktop package is only officially shipped for one OS today.

### Phase 5 — Testing depth
Add direct unit tests for `optimization.py`, `market_data.py`, and
`secrets_store.py`. Deduplicate the 102-file `ROOT = ...` boilerplate and the
duplicated `sample_config()` helper via the Phase 0 `conftest.py`. Introduce
a minimal JS test runner (Node's built-in `node:test` is sufficient — no new
dependency) for the highest-risk pure functions in `dashboard.js`, rather than
relying solely on Python-side string matching.

### Phase 6 — Usability
Accessibility pass on `index.html`/`admin.html` (labels, roles, focus
management). Write `CONTRIBUTING.md` documenting the real dev setup once
Phase 0 lands.

## 11. Recommended model and execution approach

This plan is large enough, and touches security-sensitive and
architecturally significant code (Phase 2's module splits, Phase 0's
security tests) closely enough, that a single flat pass with one model
setting is the wrong shape. Recommendation:

- **Claude Opus 4.8**, run at `effort: high` (or `xhigh` for the hardest
  single steps), for: designing and executing Phase 2's module
  decompositions, the Phase 0 security-module tests, and the
  `plan_routes.py`/`app_core.py` service-extraction completion. These need
  the strongest available reasoning because a bad split silently changes
  behavior in ways tests may not catch, and because auth/audit code is the
  worst place in the repo to introduce a regression.
- **Claude Sonnet 5** for the high-volume, well-specified mechanical work
  once Phase 0/Phase 2 establish the pattern: writing the bulk of new unit
  tests that follow an established template, deleting confirmed-dead code,
  renaming files, fixing the `tempfile`/temp-filename bugs, and the
  accessibility pass. Sonnet 5 is near-Opus quality on coding/agentic work at
  roughly half the cost — appropriate once a human or Opus has set the
  pattern to follow.
- Run each phase as its own branch/PR gated by the Phase-0 CI once it
  exists; don't batch Phase 2's module splits into one giant diff — the repo
  has no lockfile or CI today, so the first PR (Phase 0) is the one that
  should get the most scrutiny, since everything after it leans on it for
  safety.
